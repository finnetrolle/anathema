# Finding 2: Dashboard Query Scope and Scaling

## Проблема

`loadTimelineDashboard()` сначала читает справочные данные, а затем тянет из БД
все не-epic issues со связями и обрабатывает их в памяти.

Затронутые места:

- `src/modules/timeline/load-dashboard.ts`
- `src/modules/timeline/build-timeline.ts`
- `prisma/schema.prisma`

Симптомы текущего подхода:

- стоимость запроса растет вместе со всем объемом Jira-данных, а не только с
  видимым диапазоном;
- в память поднимается больше данных, чем реально нужно для текущего экрана;
- `rawPayload` сериализуется и парсится даже для задач, которые пользователь не
  увидит в текущем диапазоне.

## Почему это важно

На MVP с десятками задач это почти незаметно, но на живом проекте с сотнями или
тысячами issues сервер начнет тратить время на:

- лишнее чтение из Postgres;
- лишние join/include;
- лишнюю JSON-обработку;
- лишнюю сериализацию на границе server/client.

Это будет напрямую бить по latency страницы.

## Целевое поведение

Стоимость запроса должна зависеть от **видимого диапазона и выбранного
проекта**, а не от общего размера базы.

Идеальная модель:

- диапазон вычисляется до запроса в БД;
- в SQL попадают только пересекающиеся с диапазоном задачи;
- из БД читаются только нужные поля;
- тяжелые derived-поля денормализуются или считаются только для реально видимых
  issues.

## Рекомендуемое решение

Рекомендуемый путь: **range-first loading**.

### 1. Сначала вычислять диапазон

Сейчас диапазон формируется внутри `buildTimelineModel()`, после загрузки всех
данных. Это нужно развернуть:

1. в `loadTimelineDashboard()` вычислить фактический `visibleStart` и
   `visibleEnd`;
2. передать их в БД-запрос;
3. уже потом строить таймлайн из ограниченного набора задач.

### 2. Фильтровать по пересечению диапазона на уровне БД

Нужен запрос вида:

- issue не epic;
- проект совпадает, если выбран фильтр;
- задача пересекается с видимым окном.

Логика пересечения:

- `markerAt >= visibleStart`
- `COALESCE(startedAt, markerAt) <= visibleEnd`

В Prisma это можно выразить через `OR`, если без raw SQL:

- ветка для задач со `startedAt`;
- ветка для задач без `startedAt`, где ориентиром служит `markerAt`.

### 3. Уменьшить payload из БД

Вместо `include` всего объекта читать только реально нужные поля через `select`.

Минимальный набор:

- поля issue для таймлайна;
- `epic.key`, `epic.summary`;
- `assignee.displayName`, `assignee.color`;
- `project.connection.baseUrl`;
- `rawPayload` только если пока нельзя быстро отказаться от него.

### 4. Снизить зависимость от `rawPayload`

Следующий шаг после оптимизации запроса:

- вынести часто используемые derived-данные в отдельные поля таблицы `Issue`;
- не парсить огромный JSON на каждый экран.

Кандидаты на денормализацию:

- `componentName`
- `authorName`
- `estimateHours`
- `estimateStoryPoints`
- development summary (`pullRequestStatus`, `pullRequestCount`, `commitCount`)

## Какие изменения нужны

### В коде

- `src/modules/timeline/load-dashboard.ts`
  - вынести вычисление диапазона выше;
  - изменить Prisma-запрос на range-aware;
  - перейти с `include` на `select`;
- `src/modules/timeline/build-timeline.ts`
  - принимать уже вычисленный диапазон или работать с готовыми `visibleStart`
    / `visibleEnd`;
- при необходимости вынести общую логику диапазона в отдельный helper.

### В схеме

Желательные индексы:

- составной индекс по `jiraProjectId`, `issueType`, `startedAt`, `markerAt`
- возможно отдельный индекс по `issueType`, `markerAt`

Точный набор стоит выбрать после `EXPLAIN ANALYZE`.

## Пошаговый план внедрения

1. Вытащить расчет диапазона из `buildTimelineModel()` в reusable helper.
2. Обновить `loadTimelineDashboard()` так, чтобы он сначала знал диапазон.
3. Ограничить SQL-запрос по диапазону и проекту.
4. Переключить чтение на `select`.
5. Померить latency до и после.
6. Затем отдельно решить, какие derived-поля денормализовать.

## Критерии готовности

- время ответа страницы растет вместе с количеством видимых задач, а не всей БД;
- в запросах нет чтения всех issues "на всякий случай";
- память процесса заметно не растет при увеличении общего объема исторических
  данных;
- построение борда на одном проекте не трогает несвязанные проекты.

## Что не надо делать

- не стоит сразу тащить сложный кэш, пока не исправлен сам запрос;
- не стоит денормализовывать все подряд без измерений;
- не стоит держать `rawPayload` как единственный источник данных для часто
  используемых полей.
