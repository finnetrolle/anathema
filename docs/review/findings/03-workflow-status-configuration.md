# Finding 3: Workflow Semantics Must Be Configurable

## Проблема

Логика `isInProgressStatus()`, `isDoneStatus()` и `deriveStartedAt()` сейчас
строится на жестко зашитых строковых списках статусов:

- `In Progress`
- `Development`
- `Code Review`
- `Done`
- `Closed`
- `Resolved`
- и т.д.

Затронутые места:

- `src/modules/jira/derive.ts`
- `src/modules/timeline/load-dashboard.ts`
- `prisma/schema.prisma`

Такой подход работает только для одного семейства workflow. Любая кастомизация
или локализация Jira-статусов ломает смысл аналитики.

## Почему это важно

От этих правил зависят ключевые поля:

- `startedAt`
- `isCompleted`
- `isMissingDueDate`
- `markerKind`

Если статусы интерпретированы неверно, продукт будет показывать не просто
ошибочные подписи, а неверную операционную картину команды.

## Целевое поведение

Правила интерпретации workflow должны быть привязаны к конкретному Jira
подключению и должны настраиваться без правки кода.

Минимум, который нужен:

- список статусов "в работе";
- список статусов "завершено";
- fallback-значения по умолчанию для пустой конфигурации.

## Рекомендуемое решение

Рекомендуемый путь: **connection-scoped workflow config**.

### 1. Хранить правила рядом с `JiraConnection`

Самый практичный MVP-вариант:

- добавить в `JiraConnection` поле `workflowRules Json?`

Пример структуры:

```json
{
  "inProgressStatuses": ["In Progress", "Code Review", "QA"],
  "doneStatuses": ["Done", "Closed", "Resolved"]
}
```

Если потребуется больше строгости, позже можно вынести это в отдельную таблицу.

### 2. Передавать правила в derive-функции

Нужно уйти от глобальных `Set` внутри `derive.ts` к функциям вида:

- `isInProgressStatus(status, rules)`
- `isDoneStatus(status, rules)`
- `deriveTimelineFields(issue, rules)`

### 3. Использовать status category там, где это возможно

Если Jira-ответ уже содержит `statusCategory`, его стоит использовать как
первичный сигнал для текущего статуса:

- `done`
- `indeterminate`
- `new`

Но для changelog этого обычно недостаточно, потому что в истории переходов у нас
есть строковые названия. Поэтому конфиг все равно нужен.

### 4. Добавить разумный fallback

Если для подключения правила не заданы:

- использовать текущие дефолтные списки;
- логировать предупреждение, что инстанс работает в fallback-режиме.

## Какие изменения нужны

### В схеме

- `prisma/schema.prisma`
  - добавить `workflowRules Json?` в `JiraConnection`

### В коде

- `src/modules/jira/derive.ts`
  - убрать жесткую привязку к глобальным `Set`;
  - принимать конфиг явно;
- `src/modules/jira/persist.ts`
  - при расчете `startedAt` и `markerKind` использовать правила подключения;
- `src/modules/timeline/load-dashboard.ts`
  - при расчете `isCompleted` и `isMissingDueDate` использовать те же правила;
- при желании добавить env bootstrap для первичной настройки.

## Пошаговый план внедрения

1. Добавить `workflowRules` в Prisma-модель.
2. Описать TypeScript-тип для правил workflow.
3. Протащить правила от `JiraConnection` до derive-функций.
4. Добавить fallback к текущим дефолтным спискам.
5. Написать тесты на англоязычный, локализованный и кастомный workflow.

## Критерии готовности

- один и тот же код корректно работает с разными Jira workflow;
- правила можно сменить без редактирования `derive.ts`;
- локализованные статусы не ломают `startedAt` и `isCompleted`;
- fallback сохраняет обратную совместимость для существующей инсталляции.

## Полезное расширение на будущее

После базового решения можно научить sync автоматически предлагать стартовую
конфигурацию:

- подтягивать список статусов из Jira;
- раскладывать их по status category;
- сохранять черновик правил для ручной проверки.
