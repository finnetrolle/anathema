# Finding 4: Timeline Must Respect Connection Timezone

## Проблема

В `JiraConnection` уже хранится `timezone`, но построение таймлайна сейчас
ориентируется на UTC и локальное серверное время:

- форматтеры созданы с `timeZone: "UTC"`;
- "сегодня" вычисляется через `new Date()`;
- парсинг `YYYY-MM-DD` делается как UTC-дата;
- дефолтный диапазон стартует от "текущей недели" без учета timezone
  подключения.

Затронутые места:

- `src/modules/timeline/build-timeline.ts`
- `src/modules/timeline/load-dashboard.ts`
- `prisma/schema.prisma`
- частично `src/modules/jira/persist.ts`

## Почему это важно

Для задач на границах суток ошибка timezone меняет саму смысловую картину:

- задача может попасть не в тот день;
- today-highlight будет сдвинут;
- пользователь увидит неправильное окно "текущей недели";
- due date без времени может визуально сместиться.

Если таймлайн служит инструментом ежедневного планирования, это уже
пользовательский дефект, а не просто техническая неточность.

## Целевое поведение

Все операции уровня календарного дня должны учитывать timezone конкретного Jira
подключения:

- форматирование дат;
- определение today;
- старт текущей недели;
- интерпретация `YYYY-MM-DD` из query params;
- сравнение day-slot'ов на борде.

## Рекомендуемое решение

Рекомендуемый путь: **вынести календарную логику в timezone-aware helper**.

### 1. Передавать timezone явно

`buildTimelineModel()` должен получать `timezone` как часть входных параметров.

То же значение должно приходить из `JiraConnection.timezone` через
`loadTimelineDashboard()`.

### 2. Убрать UTC-утилиты общего назначения

Сейчас в `build-timeline.ts` функции вроде `floorToDay()`, `dayKey()` и
`startOfCurrentWeek()` живут в UTC-модели.

Их нужно заменить на helper-слой, который работает в заданной timezone:

- `getDayKey(date, timezone)`
- `getStartOfDay(date, timezone)`
- `getStartOfWeek(date, timezone)`
- `parseDateInputInTimezone(value, timezone)`

### 3. Использовать библиотеку для timezone-арифметики

Для этой задачи лучше не писать собственную арифметику на `Date`, потому что
ошибки быстро появятся на DST и границах суток.

Практичный вариант:

- добавить `date-fns-tz` или `luxon`

Это упростит:

- перевод даты в нужную временную зону;
- начало дня/недели;
- форматирование для UI.

### 4. Отдельно проверить due date без времени

Сейчас `parseJiraDate()` превращает date-only значение в `T12:00:00.000Z`.

Это допустимый технический трюк, но его нужно согласовать с новой моделью:

- либо хранить due date как "локальный день" и рендерить через timezone helper;
- либо документировать, что полдень UTC используется только как безопасный
  носитель календарной даты.

Важно, чтобы эта договоренность была одной и той же в persistence и в render.

## Какие изменения нужны

### В коде

- `src/modules/timeline/build-timeline.ts`
  - убрать module-level `Intl.DateTimeFormat` с UTC;
  - создавать formatter'ы под конкретную timezone;
  - переписать day math через общий helper;
- `src/modules/timeline/load-dashboard.ts`
  - передавать timezone в builder;
- возможно добавить новый модуль, например
  `src/modules/timeline/date-helpers.ts`.

### В схеме и данных

- использовать `JiraConnection.timezone` как источник истины;
- при нескольких подключениях заранее решить правило:
  один борд = одна timezone, либо нужна агрегация по подключениям.

## Пошаговый план внедрения

1. Добавить отдельный timezone-aware helper.
2. Протащить timezone из `JiraConnection` в `buildTimelineModel()`.
3. Переписать `dayKey`, `today`, `rangeStart`, `rangeEnd`, label formatting.
4. Отдельно проверить сценарии с `dueAt` и `startedAt` на границах суток.
5. Добавить тесты для минимум двух timezone, например `Europe/Moscow` и
   `America/Los_Angeles`.

## Критерии готовности

- одна и та же задача попадает в одинаковый календарный день во всех частях UI;
- today-highlight соответствует timezone подключения;
- выбор диапазона `from` / `to` интерпретируется в той же timezone;
- тесты проходят на датах вокруг полуночи и вокруг переходов DST.

## Важное архитектурное замечание

Если продукт в будущем будет показывать данные из нескольких Jira-подключений на
одном экране, timezone нужно сделать явной частью view-model, а не молчаливым
глобальным предположением.
