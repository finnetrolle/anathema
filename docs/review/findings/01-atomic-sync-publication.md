# Finding 1: Atomic Sync Publication

## Проблема

Сейчас `runJiraSyncChunk()` пишет данные прямо в основные таблицы
`Issue`, `Epic`, `Assignee` и `IssueStatusHistory` по мере прихода чанков, а
успешность `SyncRun` определяется только в конце полного прогона.

Затронутые места:

- `src/modules/jira/persist.ts`
- `src/modules/timeline/load-dashboard.ts`

Это означает, что:

- отмененный sync успевает частично обновить живые данные;
- упавший sync оставляет смесь старого и нового состояния;
- UI не умеет отличать "полностью опубликованный снимок" от "данных, которые
  еще грузятся или уже упали".

## Почему это важно

Текущий дефект бьет по самой ценной части продукта: достоверности таймлайна.
Пользователь видит корректно отрендеренный интерфейс, но данные внутри него
могут быть логически несогласованными.

Это хуже, чем явная ошибка:

- нельзя доверять аналитике после неуспешного sync;
- сложно расследовать инциденты, потому что таблицы уже изменены;
- повторный sync не всегда возвращает систему в предсказуемое состояние.

## Целевое поведение

Борд должен читать только полностью опубликованный снимок данных.

Во время sync возможны два безопасных состояния:

1. UI продолжает показывать последний успешный снимок.
2. Новый снимок готовится в фоне и становится видимым только после полного
   завершения sync.

## Рекомендуемое решение

Рекомендуемый путь: **staging + publish**.

### Идея

1. Каждый `SyncRun` пишет данные не в боевые таблицы, а в staging-слой,
   привязанный к `syncRunId`.
2. После получения последнего чанка приложение в одной транзакции:
   - проверяет, что sync завершен успешно;
   - очищает текущий опубликованный набор для нужного подключения;
   - переносит staging-данные в боевые таблицы;
   - помечает `SyncRun` как `SUCCEEDED`.
3. Если sync отменен или упал, staging-данные можно удалить без влияния на UI.

### Почему этот вариант лучше остальных

- легко объясняется и поддерживается;
- не требует читать "последний успешный run" из каждой рабочей таблицы;
- позволяет оставить текущую read-модель простой;
- хорошо сочетается с chunked import.

## Какие изменения нужны

### 1. Добавить staging-таблицы

Минимальный набор:

- `StagedEpic`
- `StagedAssignee`
- `StagedIssue`
- `StagedIssueStatusHistory`

Обязательные поля:

- `syncRunId`
- естественный Jira-идентификатор (`jiraIssueId`, `jiraEpicId`, `jiraAccountId`)
- все поля, которые сегодня пишутся в боевые таблицы

### 2. Перенести запись чанков в staging

В `persistIssues()` заменить upsert в боевые таблицы на upsert в staging-таблицы.

Принцип:

- каждый чанк идемпотентен в рамках одного `syncRunId`;
- повторный вызов того же чанка не должен плодить дубли;
- уникальные индексы в staging должны включать `syncRunId`.

### 3. Добавить фазу публикации

После последнего чанка нужен отдельный шаг `publishSyncRun(syncRunId)`.

Шаг должен выполняться в одной транзакции:

1. Проверить, что все staging-данные принадлежат нужному `jiraConnectionId`.
2. Удалить или заменить текущие опубликованные записи для этого подключения.
3. Переложить staging-данные в боевые таблицы.
4. Обновить `SyncRun.status = SUCCEEDED` и `finishedAt`.

### 4. Очистка после ошибок

При `FAILED` или aborted sync:

- пометить `SyncRun` как `FAILED`;
- удалить staging-данные этого прогона;
- не менять опубликованный снимок.

## Точки изменения в коде

- `src/modules/jira/persist.ts`
  - вынести запись чанка в `persistIssuesToStage()`;
  - добавить `publishSyncRun()`;
  - вызывать публикацию только на финальном чанке;
- `prisma/schema.prisma`
  - описать staging-таблицы и индексы;
- `src/modules/timeline/load-dashboard.ts`
  - в идеале без изменений, если UI продолжит читать только боевые таблицы.

## Пошаговый план внедрения

1. Добавить staging-модели в Prisma.
2. Реализовать запись чанков в staging.
3. Реализовать публикацию в транзакции.
4. Переключить `runJiraSyncChunk()` на новую схему.
5. Добавить cleanup для failed/aborted sync.
6. Написать интеграционный тест на сценарий "упал на втором чанке".

## Критерии готовности

- после failed sync борд показывает данные предыдущего успешного прогона;
- после aborted sync боевые таблицы не меняются;
- после успешного полного sync UI видит новый снимок целиком;
- повторный запуск того же sync не приводит к дублям.

## Альтернатива

Альтернативный вариант: versioned snapshot в основных таблицах
(`snapshotId`/`syncRunId` + чтение только последнего успешного снимка).

Он тоже корректен, но дороже в чтении и сложнее в сопровождении, если сейчас не
нужна история снимков прямо в боевых таблицах.
